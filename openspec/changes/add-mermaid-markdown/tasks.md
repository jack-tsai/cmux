## 3. MermaidBlockView

- [x] 3.1 新增 `Sources/Panels/MermaidBlockView.swift` 為 `NSViewRepresentable`（包在外層 `MermaidBlockView: View` 內）、每段 mermaid 對應一個 `WKWebView` 實例；以 `loadHTMLString(html, baseURL: mermaidDir)` 載入（`baseURL` 指向 bundle 裡的 `Mermaid/` folder reference，確保 `mermaid.min.js` 解析為同源）；對應 design 決策「每個 mermaid block 一個獨立 `WKWebView`」，並實作 spec 需求 Render mermaid code blocks as diagrams in the Markdown panel。
- [x] 3.2 在 `WKWebViewConfiguration.userContentController` 以 `Coordinator` 註冊 `cmuxMermaid` message handler；`userContentController(_:didReceive:)` 解碼 `{type, height, error, kind, message}` 後以 `DispatchQueue.main.async` 更新 `@State`（height via `onRendered` closure、error via `onError` closure），未在 SwiftUI body 內 mutate state；對應 design 決策「以 JS→Swift Bridge 回報 intrinsic height」。
- [x] 3.3 收到 `{type:"rendered", height}` 後更新 `@State measuredHeight` 驅動 `.frame(height: measuredHeight)`；`height` 同時以 `Double` / `Int` 解包 fallback；由於 template 用 `Math.ceil(document.body.scrollHeight)` 量測，無巢狀 scrollbar；實作 spec 需求 Mermaid diagram size SHALL match its rendered content。
- [x] 3.4 外層 `MermaidBlockView` observe `@Environment(\.colorScheme)`，colorScheme 變動時經 `updateNSView` 進入 `Coordinator.applyUpdate` → `evaluateJavaScript("window.cmuxMermaidRender(source, theme)")`（template 該 API 同時接受新 theme 並 re-render），WebView 實例不重建、SwiftUI 不重掛；對應 design 決策「Dark / Light 以 `mermaid.initialize({ theme })` + `mermaid.run()` 切換，不重建 WebView」並實作 spec 需求 Mermaid rendering SHALL follow the active color scheme。
- [x] 3.5 `source: String` 變動時同樣走 `updateNSView` → `evaluateJavaScript("window.cmuxMermaidRender(newSource, currentTheme)")`，不重建 WebView；對應 design 決策「內容變更（`panel.content` 改變）時以 token 觸發 re-render，不換 WebView」。
- [x] 3.6 實作錯誤態：(a) template.html `window.addEventListener("error", ...)` 捕捉 `mermaid.min.js` 載入失敗 → bridge `{type:"error", kind:"runtimeMissing"}`；(b) Coordinator `webView(_:didFailProvisionalNavigation:withError:)` 與 `webView(_:didFail:withError:)` 走同路徑 → `.runtimeMissing`；(c) `didFinishNavigation` / 每次 update 都重啟 `startWatchdog()`（15 s `DispatchWorkItem`）— 未在 timeout 內收到任何 bridge 訊息則觸發 `.runtimeMissing`；bridge 送來 `{kind:"syntax"}` 走 `.syntax`；錯誤 UI 為 `MermaidInlineErrorView`（`exclamationmark.triangle.fill` + 紅色邊框 + localized prefix + 原始 monospaced source），不彈 alert、不影響其他區塊；對應 design 決策「語法錯誤就地顯示紅底錯誤訊息」並實作 spec 需求 Mermaid syntax errors SHALL be reported in-place。
- [x] 3.7 在 `Resources/Localizable.xcstrings` 新增 5 個 keys (`mermaid.error.title`, `mermaid.error.syntax`, `mermaid.error.runtimeMissing`, `mermaid.error.timeoutMessage`, `mermaid.view.loading`)，每個都填 en + ja 翻譯；所有錯誤文字以 `String(localized:)` 呼叫。

- [x] 6.2 補建 `cmuxTests/MermaidRendererTests.swift`（task 2.1 / proposal impact 原本宣稱要新增但實際遺漏）；測試 escape 的 JSON-literal 輸出 / `</` 中和 / LS/PS 編碼 / 惡意 source 不脫離字串字面量、`apply` 的 placeholder 替換、`MermaidTheme` 映射、以及空 bundle 時 fallback 行為。加入 pbxproj cmuxTests target（IDs `A5003200` / `A5003201`）。**測試跑起來後抓到實作 bug（見 6.4）**。
- [x] 6.4 修正 `MermaidRenderer.escape` 的 double-wrap bug：`JSONSerialization.data(withJSONObject: [source])` 輸出 `["..."]`，原本只 drop 首尾一字元得到 `"..."`（**含**引號），最後又被 `"\"\(encoded)\""` 再包一層變成 `""..."" ` — 產生的 JS 字串字面量是 `""x""`（語法錯誤），讓 template 的 `var currentSource = {{SOURCE}};` 與 `window.cmuxMermaidRender(\(sourceLiteral), ...)` 都會爛掉。改成 `dropFirst(2).dropLast(2)` 把 `["` 與 `"]` 一起剝掉，讓 JSON path 與 manualEscape path 都只輸出 inner 內容，最後統一外層補一對引號。這個 bug 若只修 loadFileURL 也還是會讓 render 失敗，屬於連環 bug 的第二層；由 task 6.2 新測試揪出。
- [x] 6.3 `scripts/ensure-ghosttykit.sh` 支援 `CMUX_SKIP_ZIG_BUILD=1`：當 zig host 版本與 ghostty submodule pin 版本不相容時（例如 zig 0.16 vs 要求 0.15.2），只要 local xcframework 存在就直接 reuse，不因 build key mismatch 觸發 rebuild。對應 reload 流程在 macOS + 新 zig 的可用性。
